Incident Response Playbook: From Detection to Recovery

By Rehan March 28, 2024 12 min read

Why Every Organization Needs an IR Plan

Cyber threats are no longer a matter of if but when. A structured incident response (IR) framework minimises downtime, data loss, and financial damage. In this guide, we dissect the SANS six-step methodology and translate it into actionable steps – whether you're a SOC analyst or a startup founder. No fluff, only playbooks.

The Six Phases of Incident Response

1. Preparation – The foundation

Most breaches succeed because of inadequate preparation. This phase is continuous: tools, training, and threat intelligence.

  • Asset inventory: Know what you protect (hardware, software, data).
  • IR toolkit: Forensic imagers (FTK Imager), EDR (CrowdStrike, SentinelOne), log aggregators (Splunk, ELK).
  • Communication matrix: Internal (CISO, legal, PR) and external (law enforcement, regulators).
  • Tabletop exercises: Simulate ransomware, BEC, or insider threat quarterly.

2. Identification – Who, what, when?

Alert triage is an art. Don't chase false positives; validate with telemetry.

Common detection sources:

  • EDR alerts (unusual parent-child processes, LSASS access)
  • User reports (phishing, unusual account behaviour)
  • Network anomalies (beaconing, rare egress traffic)
  • Third-party notification (CERT, dark web monitoring)

Key question: Is this a security incident or a false alarm? Use the Pyramid of Pain to assess severity.

3. Containment – Stop the bleeding

Short-term containment vs. long-term remediation. Speed matters, but so does evidence preservation.

privilege::debug

  • Network isolation: move compromised host to quarantine VLAN, block C2 IPs at firewall.
  • Endpoint: disable local admin, reset service account credentials.
  • Cloud: revoke temporary OAuth tokens, restrict IAM roles.

4. Eradication – Remove the adversary

Eliminate persistence, backdoors, and rootkits. Don't forget cloud artifacts (unused Lambda functions, compromised API keys).

  • Reimage endpoints – never trust a compromised host.
  • Rotate all secrets in the blast radius.
  • Patch the root vector (unpatched vulnerability, misconfiguration).

5. Recovery – Back to business

Restore systems cautiously. Monitor for lingering indicators.

  • Restore from clean, verified backups (air‑gapped if possible).
  • Increase logging verbosity during restoration phase.
  • Phased roll‑out to production – canary deployments.

6. Lessons Learned – Close the loop

The post‑mortem is not a blame session. Update runbooks, improve detection, and adjust budgets.

Real‑world IR: Ransomware Playbook

Scenario: Cobalt Strike beacon followed by LockBit 3.0 encryption.

  1. Detect: Splunk alert – high volume of SMB writes + event ID 5145 (network share object access).
  2. Contain: EDR – kill process, firewall block domain controller outbound, disable AD sync.
  3. Eradicate: Scan for scheduled tasks, WMI persistence. Reset krbtgt password twice.
  4. Recover: Restore from immutable backups (Veeam). Deploy EDR on all domain controllers.

Metrics That Matter

  • MTTD (Mean Time to Detect) – target < 1 hour.
  • MTTC (Mean Time to Contain) – ideal < 30 minutes.
  • IR coverage % – percentage of assets covered by EDR.
  • False positive rate – tune after each campaign.

Common Pitfalls & How to Avoid Them

  • No legal hold: Ensure chain of custody – use write blockers.
  • Isolating without preserving memory: Capture RAM before shutdown.
  • Over‑relying on automation: Human validation still required.
  • Ignoring insider threats: Monitor DLP, UEBA.

Certifications & Training Paths

If you're building an IR career: SANS FOR508, GIAC GCIH, BTL1, or CREST CPSA. Also, contribute to open‑source DFIR projects (Plaso, Timesketch).

Conclusion

Incident response is a discipline, not a tool. Invest in preparation, practice muscle memory, and communicate clearly. Every breach is a learning opportunity – treat it as one. Bookmark this page; you might need it at 3 AM.
