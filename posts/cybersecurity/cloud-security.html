Cloud Security 2024: IAM, Zero Trust & Shared Responsibility

By Rehan February 12, 2025 12 min read

Introduction

As organizations accelerate cloud adoption, securing infrastructure, applications, and data in AWS, Azure, and GCP has never been more critical. Cloud security is not just the provider's responsibility — it demands a shared mindset, robust Identity and Access Management (IAM), and continuous compliance. This guide breaks down the core pillars of modern cloud security, including real-world misconfigurations, hands-on IAM hardening, and Zero Trust architecture. Whether you're a cloud engineer or a security analyst, you'll walk away with actionable defenses for your cloud environment.

1. The Shared Responsibility Model (Decoded)

One of the most misunderstood concepts in cloud security. Let's clarify: the cloud provider secures of the cloud; you secure in the cloud.

Who is responsible for what?

  • AWS/Azure/GCP: Physical security, network infrastructure, virtualization layer, hardware.
  • You (the customer): Identity & access, data encryption (at rest/in transit), OS patches, firewall rules, network ACLs, application configurations.
  • Common pitfall: Leaving S3 buckets or Azure Blob containers publicly writable — 90% of cloud breaches involve customer misconfiguration (Cloud Security Alliance).
# Example: Publicly exposed AWS S3 bucket policy (DON'T DO THIS)
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::confidential-bucket/*"
}
// ❌ This grants anonymous read access to everyone.

2. Identity and Access Management (IAM) Hardening

Weak IAM is the root cause of 70% of cloud data breaches. Enforce least privilege, use temporary credentials, and audit continuously.

Critical IAM Controls

  • No root user: Enable MFA, remove access keys, delegate via roles.
  • Use IAM roles instead of long-lived keys: EC2 instance profiles, OIDC federation for Kubernetes.
  • Permission boundaries & SCPs: Service control policies in AWS Organizations to restrict even admins.
  • Conditional access (Azure AD): Risk-based sign-in, device compliance.
// Secure IAM role trust policy (AWS)
{
  "Effect": "Allow",
  "Principal": {
    "Federated": "accounts.google.com"
  },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": {
      "token.actionscript.us:sub": "service-account@project.iam.gserviceaccount.com"
    }
  }
}

3. Zero Trust Network Security

Castle-and-moat is dead. Assume breach, verify explicitly, and segment everything.

Practical Zero Trust in the cloud:

  • Micro-segmentation: Use Kubernetes Network Policies, AWS Security Groups (stateful), Azure NSGs.
  • No direct internet access: VPC peering, PrivateLink, or Google Cloud VPC SC.
  • Cloud WAF & DDoS protection: AWS WAF, Cloud Armor, Azure DDoS Protection.
  • Secure endpoints: BeyondCorp / Zero Trust proxies (Zscaler, Cloudflare Access).
# Kubernetes NetworkPolicy – deny all ingress except from monitoring
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-external
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: observability

4. Encryption & Key Management

Data at rest and in transit must be encrypted — but proper key management is non-negotiable.

Common pitfalls

  • Using default KMS keys without rotation.
  • Embedding secrets in code or environment variables (use Hashicorp Vault / AWS Secrets Manager).
  • Disabling TLS 1.2/1.3 on load balancers.

Best practices:

  • Enforce encryption at rest: S3-SSE, EBS volumes, RDS encryption.
  • Customer-managed keys (CMK): With automatic rotation (AWS KMS, Azure Key Vault, GCP Cloud KMS).
  • Secrets management: Inject via CSI driver or external secrets operator.
// Terraform: AWS S3 bucket with enforced encryption
resource "aws_s3_bucket" "secure_bucket" {
  bucket = "rehan-secure-logs"
  
  server_side_encryption_configuration {
    rule {
      apply_server_side_encryption_by_default {
        sse_algorithm = "aws:kms"
        kms_master_key_id = aws_kms_key.bucket_key.arn
      }
    }
  }
}

5. Cloud Security Posture Management (CSPM)

Continuous compliance monitoring against benchmarks (CIS, NIST, PCI-DSS). Manual reviews are dead — automation is the only way.

Tools & tactics:

  • Native tools: AWS Config, Azure Security Center, GCP Security Command Center.
  • Open source: Prowler, ScoutSuite, Checkov (IaC scanning).
  • Remediation: Auto-remediate non-compliant resources via EventBridge / Cloud Functions.
# Prowler command – CIS benchmark check
prowler aws --checks check_s3_bucket_public_read,check_iam_root_mfa
# Outputs violations and fixes in CLI or CSV

6. Container & Kubernetes Security

Containers add new layers, but also new risks. Image vulnerabilities, excessive privileges, and runtime threats.

Attack chain prevention:

  1. Build phase: Scan images with Trivy, Grype, or ECR scan; use minimal base images (distroless).
  2. Deploy phase: Admission controllers (OPA/Gatekeeper, Kyverno) enforce no privileged containers, read-only root.
  3. Runtime: Falco, Cilium Tetragon for behavioral detection.
# OPA Gatekeeper policy: block privileged containers
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sNoPrivileged
metadata:
  name: no-privileged-containers
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    exemptImages: ["gcr.io/google-containers/pause"]

7. Incident Response in the Cloud

IR shifts when you don’t own the physical servers. You need cloud-native forensics, and rapid isolation.

IR playbook example (AWS):

  1. Detect: GuardDuty finding (crypto miner, backdoor).
  2. Isolate: Detach instance from security group, remove IAM role, take EBS snapshot.
  3. Investigate: CloudTrail logs, VPC Flow Logs, Inspector agent findings.
  4. Recover: Restore clean AMI, rotate keys.
# AWS CLI: isolate compromised EC2 instance
aws ec2 modify-instance-attribute --instance-id i-0abc123 --groups sg-isolated
aws iam remove-role-from-instance-profile --instance-id i-0abc123 --role compromised-role

8. Compliance & Regulations

GDPR, HIPAA, SOC2, FedRAMP – cloud adds complexity to audits.

  • Data residency: Keep data in specific regions (e.g., eu-central-1 for GDPR).
  • Log retention: Enable CloudTrail / Azure Log Analytics with immutability.
  • Third-party audits: Use AWS Artifact or Azure Service Trust Portal for certifications.

Case Study: Capital One Breach (2019)

A WAF misconfiguration allowed SSRF, leading to stolen credentials. Root cause: overly permissive IAM role attached to WAF.

Prevention today:

  • Metadata service v2 (IMDSv2) required.
  • IAM conditions: aws:SourceIp, aws:ViaAWSService.
  • VPC endpoints for S3/SSM, no internet gateway.

Automation & DevSecOps

Shift left security: Scan Terraform, CloudFormation, and manifests before deployment.

  • Checkov / tfsec for misconfigurations (e.g., S3 bucket logging disabled).
  • Pre-commit hooks for secret detection (detect-secrets, truffleHog).
  • CI/CD guardrails: GitHub Advanced Security, GitLab SAST.
# Checkov: detect S3 without versioning
checkov -d . --framework terraform
# Pass: Ensure S3 bucket has versioning enabled (CKV_AWS_21)

Looking Ahead: AI & Cloud Security

  • CSPM+AI: Predictive analytics for privilege escalation.
  • LLM-based log analysis: Amazon Q, Microsoft Security Copilot.
  • Confidential computing: Always encrypted in use (AMD SEV-SNP, Intel TDX).

Conclusion

Cloud security in 2025 is a shared, automated, and continuous process. The fundamentals remain: lock down IAM, encrypt everything, monitor posture, and assume breach. Start by enabling CSPM tools, removing public access, and implementing Zero Trust principles. The cloud can be more secure than on-prem — if you follow the blueprint above.

Ready to harden your cloud? Audit your environment with Prowler today, and revisit your IAM policies. Your future self (and security team) will thank you.
