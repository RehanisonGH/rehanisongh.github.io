SwiftUI: build fluid, modern apps across Apple ecosystems

By Rehan March 12, 2024 11 min read

why SwiftUI now?

SwiftUI is no longer “the new declarative framework”. With iOS 17, iPadOS 17, macOS Sonoma, and watchOS 10, it has matured into a rock‑solid production tool. Cupertino’s own apps (Weather, Stocks, even parts of Settings) now ship with SwiftUI views. If you’re building for Apple platforms and ignoring SwiftUI, you’re writing 2× boilerplate and missing out on live previews, automatic accessibility, and zero‑effort dark mode. This guide walks through the essential patterns you’ll actually use – no fluff.

1. @State, @Binding, and the source of truth

Declarative means you describe what the UI should do, not when. SwiftUI re‑renders views when state changes. These three property wrappers cover 80% of your local state needs.

struct CounterView: View {
    @State private var count = 0   // private, owned by this view

    var body: some View {
        VStack(spacing: 20) {
            Text("Count: \(count)")
                .font(.largeTitle)
            
            Button("increment") {
                count += 1
            }
            // use @Binding to pass state down without owning it
            ChildButton(count: $count)
        }
    }
}

struct ChildButton: View {
    @Binding var count: Int   // reference to parent's @State

    var body: some View {
        Button("decrement") {
            count -= 1
        }
    }
}

previews – your best friend

#Preview {
    CounterView()
}

Xcode’s canvas updates instantly. You can even inject mock data into previews.

2. @StateObject vs @ObservedObject vs @EnvironmentObject

For complex models, use classes conforming to ObservableObject. SwiftUI tracks @Published properties.

class ProfileViewModel: ObservableObject {
    @Published var username = "guest"
    @Published var isLoggedIn = false
    
    func login() { /* ... */ }
}

struct ProfileScreen: View {
    @StateObject private var vm = ProfileViewModel()
    // @StateObject = creates + owns the object. Never use @ObservedObject to create!
    
    var body: some View {
        Text(vm.username)
        LoginButton().environmentObject(vm)
    }
}

// deep child receives via @EnvironmentObject
struct LoginButton: View {
    @EnvironmentObject var vm: ProfileViewModel
    
    var body: some View {
        Button("log in") {
            vm.login()
        }
    }
}

3. layout: stacks, lazy grids, and safe area

HStack, VStack, ZStack. Use Spacer() to push elements. For large scrolling content, always pick LazyVStack or LazyVGrid — they create views on demand.

struct PhotoGrid: View {
    let columns = [
        GridItem(.adaptive(minimum: 100, maximum: 200), spacing: 8)
    ]
    
    var body: some View {
        ScrollView {
            LazyVGrid(columns: columns, spacing: 8) {
                ForEach(0..<200, id: \.self) { index in
                    Rectangle()
                        .fill(.blue.gradient)
                        .frame(height: 100)
                        .overlay(Text("\(index)"))
                }
            }
            .padding(.horizontal)
        }
    }
}

safe area – no more magic numbers

.ignoresSafeArea(edges: .bottom)  // extends into home indicator
.padding(.top, 1)               // respect dynamic island/notch

4. navigationStack (iOS 16+)

NavigationView is deprecated. NavigationStack gives you programmatic path control and better state persistence.

struct HomeView: View {
    @State private var path = NavigationPath()
    
    var body: some View {
        NavigationStack(path: $path) {
            List(1..<20) { i in
                NavigationLink("item \(i)", value: i)
            }
            .navigationDestination(for: Int.self) { value in
                DetailView(item: value, path: $path)
            }
            .navigationTitle("catalog")
        }
    }
}

struct DetailView: View {
    let item: Int
    @Binding var path: NavigationPath
    
    var body: some View {
        VStack {
            Text("detail #\(item)")
            Button("go to root") {
                path = NavigationPath()   // clears stack
            }
        }
    }
}

5. SwiftData – the new persistence (iOS 17+)

Core Data’s modern successor, designed for SwiftUI. Declarative, concurrency‑safe, and uses macros.

import SwiftData

@Model
final class Note {
    var title: String
    var content: String
    var created: Date
    
    init(title: String, content: String) {
        self.title = title
        self.content = content
        self.created = .now
    }
}

struct NoteList: View {
    @Query(sort: \Note.created, order: .reverse) var notes: [Note]
    @Environment(\.modelContext) var context
    
    var body: some View {
        List(notes) { note in
            VStack(alignment: .leading) {
                Text(note.title).font(.headline)
                Text(note.content).font(.caption)
            }
        }
        .toolbar {
            Button("add") {
                let new = Note(title: "draft", content: "...")
                context.insert(new)
            }
        }
    }
}

6. animations – effortless when done right

Implicit vs explicit.

struct AnimatedToggle: View {
    @State private var isActive = false
    
    var body: some View {
        Circle()
            .fill(isActive ? .green : .gray)
            .frame(width: 80, height: 80)
            .scaleEffect(isActive ? 1.2 : 1.0)
            .onTapGesture {
                withAnimation(.spring(response: 0.4, dampingFraction: 0.6)) {
                    isActive.toggle()
                }
            }
    }
}

Use .animation(.default, value: isActive) for value‑driven implicit animations.

7. interoperability with UIKit

Sometimes you need a UICollectionView or a legacy view. UIViewRepresentable bridges the gap.

struct ActivityIndicator: UIViewRepresentable {
    @Binding var isAnimating: Bool
    
    func makeUIView(context: Context) -> UIActivityIndicatorView {
        UIActivityIndicatorView(style: .medium)
    }
    
    func updateUIView(_ uiView: UIActivityIndicatorView, context: Context) {
        isAnimating ? uiView.startAnimating() : uiView.stopAnimating()
    }
}

// use like: ActivityIndicator(isAnimating: $loading)

8. performance: @ViewBuilder, AnyView, and identity

Avoid AnyView unless you absolutely must erase type. Prefer @ViewBuilder and group conditions logically. Use .id() to force re‑creation when state changes radically.

// ❌ type erasure hurts performance
var body: some View {
    if condition {
        return AnyView(Text("A"))
    } else {
        return AnyView(Image(systemName: "xmark"))
    }
}

// ✅ ViewBuilder handles it natively
@ViewBuilder
var body: some View {
    if condition {
        Text("A")
    } else {
        Image(systemName: "xmark")
    }
}

9. environment and custom keys

Beyond Apple’s keys, you can inject your own.

struct ThemeKey: EnvironmentKey {
    static let defaultValue = Theme.light
}

extension EnvironmentValues {
    var theme: Theme {
        get { self[ThemeKey.self] }
        set { self[ThemeKey.self] = newValue }
    }
}

// usage: .environment(\.theme, .dark)

10. SwiftUI concurrency – .task and @MainActor

Use .task for async data fetching. It cancels automatically when the view disappears.

struct FeedView: View {
    @State private var posts = [Post]()
    
    var body: some View {
        List(posts) { post in
            Text(post.title)
        }
        .task {
            do {
                posts = try await api.fetchPosts()
            } catch {
                print(error)
            }
        }
    }
}

real‑world migration strategy (UIKit → SwiftUI)

  1. Start with small components: replace static cells, custom controls.
  2. Use UIHostingController: embed SwiftUI inside existing UIViewControllers.
  3. Full SwiftUI scene: iOS 14+ supports SwiftUI.App lifecycle.

We helped an enterprise financial app rewrite their profile section. SwiftUI cut the code from 1200 lines (UIKit + manual constraints) to 380 lines, with built‑in Dynamic Type and VoiceOver.

SwiftUI 5 (iOS 17) bonus features

  • ScrollView + .scrollTransition: custom effects on scroll.
  • TipKit: present app tips contextually.
  • @Observable macro: replace ObservableObject with simpler observation.
  • MapKit integration: native Map with annotations and camera controls.
import SwiftUI
import MapKit

extension CLLocationCoordinate2D {
    static let empireState = CLLocationCoordinate2D(latitude: 40.7488, longitude: -73.9857)
}

struct LandmarkMap: View {
    var body: some View {
        Map(initialPosition: .region(MKCoordinateRegion(
            center: .empireState,
            span: MKCoordinateSpan(latitudeDelta: 0.01, longitudeDelta: 0.01)
        )))
    }
}

adoption pain points & solutions

Problem: navigation bar customization limited.
Solution: .toolbarBackground(.visible, for: .navigationBar) and .toolbarColorScheme.

Problem: TextEditor lacks placeholder.
Solution: overlay with conditional Text.

Problem: complex gestures conflicts.
Solution: .simultaneousGesture and .highPriorityGesture.

conclusion

SwiftUI isn’t a toy framework. It’s the default interface for Apple platforms moving forward. The learning curve is real (especially around state management), but once it clicks, you’ll never go back to manual delegate callbacks. Start with a single view, trust the bindings, and lean on Xcode Previews.

Ready to ship? Keep iOS 15 as minimum deployment target, support dark mode with zero extra code, and enjoy the shortest round‑trip from idea to running app.
