Mobile Testing: Complete guide for Android & iOS (2024)

By Rehan February 12, 2024 12 min read

Why mobile testing is different

Fragmentation, gestures, interrupts, network switching — mobile apps face unique challenges. With over 24k distinct Android devices and constant iOS updates, testing can’t be an afterthought. This guide walks you through modern mobile testing strategies, from emulators to real-device clouds, and from manual sanity to fully automated CI pipelines. Whether you build with Flutter, React Native, or native Kotlin/Swift, these patterns apply.

1. Testing pyramid for mobile

Unlike traditional web, mobile testing pyramid emphasizes:

  • Unit tests (60%): business logic, ViewModels, use cases (JUnit, XCTest).
  • Integration tests (25%): database, API, shared preferences (Espresso, UI Automator, XCUITest).
  • UI / end-to-end tests (15%): critical user journeys on real devices.
  • Manual & exploratory (supplement): usability, gestures, unexpected flows.

With React Native / Flutter, you also test the bridge; we’ll cover framework‑specific tools below.

Android testing toolkit

// Example: local unit test with JUnit4 + MockK
@Test
fun `login with valid credentials returns success`() {
    val mockRepo = mockk<AuthRepository>()
    coEvery { mockRepo.login("alice@example.com", "pass") } returns Result.success(User("alice"))
    val viewModel = AuthViewModel(mockRepo)
    viewModel.login("alice@example.com", "pass")
    assertEquals(viewModel.state.value, UiState.Success)
}

iOS testing with XCTest

// Unit test example Swift
func testLoginValidation() throws {
    let validator = LoginValidator()
    let isValid = validator.validate(email: "test@example.com", password: "123456")
    XCTAssertTrue(isValid, "Valid credentials should pass")
}

2. UI automation – cross‑platform vs native

For native: Espresso (Android) and XCUITest (iOS) are the gold standard. For cross‑platform: Detox (React Native) and integration_test (Flutter) offer grey‑box testing.

Detox (React Native) – grey‑box, runs on device

describe('Login flow', () => {
  it('should show home after successful login', async () => {
    await element(by.id('emailInput')).typeText('user@example.com');
    await element(by.id('passwordInput')).typeText('Pass123!');
    await element(by.id('loginButton')).tap();
    await expect(element(by.id('welcomeMessage'))).toBeVisible();
  });
});

Flutter integration test

testWidgets('failing login shows error', (tester) async {
  await tester.pumpWidget(MyApp());
  await tester.enterText(find.byKey(Key('emailField')), 'wrong');
  await tester.enterText(find.byKey(Key('passwordField')), 'wrong');
  await tester.tap(find.byKey(Key('loginBtn')));
  await tester.pumpAndSettle();
  expect(find.text('Invalid credentials'), findsOneWidget);
});

3. Real devices vs emulators / simulators

TypeProsCons
Emulator/SimulatorFree, fast, CI-friendly, easy snapshotMissing sensors, performance differences, some OEM bugs
Real device farm (AWS, Firebase, BrowserStack)True hardware, network conditions, battery, fragmentationCost, slower, flaky queues
Local real devicesAccurate, USB debuggingLimited coverage, not scalable

Recommendation: Use simulators for early development and most CI runs; run a subset of critical E2E on real devices nightly.

4. Device & OS fragmentation strategy

You don't need to test every device. Use analytics to know your top 20 models (usually cover >75% users). Prioritize:

  • Latest OS + previous 2 major versions (Android 12,13,14; iOS 16,17).
  • Common screen sizes: small (iPhone SE), medium (Pixel 6), large (Samsung S23 Ultra).
  • Low‑end devices (RAM & CPU constraints) — crucial for emerging markets.

Firebase Test Lab – matrix testing

// gcloud command to run Espresso on 10 devices in parallel
gcloud firebase test android run \
  --type instrumentation \
  --app app-debug.apk \
  --test app-debug-androidTest.apk \
  --device model=Pixel2,version=30 \
  --device model=Pixel4,version=33 \
  --device model=medium_phone,version=29 \
  --timeout 20m

5. Interrupt & lifecycle testing

Mobile apps must survive calls, notifications, low battery, and app switcher. Automate with:

  • Android: Test rules + ActivityScenario. Simulate configuration change, permission dialogs.
  • iOS: XCUIDevice.press(.home); springboard interactions.
  • React Native / Flutter: App lifecycle events via testing libraries.
// Android – simulate screen rotation
@Test
fun testRotation() {
    val scenario = ActivityScenario.launch(MainActivity::class.java)
    scenario.recreate()
    // verify state restored
}

6. Performance & memory testing

Slow UI renders, memory leaks, and battery drain are top reasons for uninstalls.

Android: Macrobenchmark (Jetpack)

@RunWith(AndroidJUnit4::class)
class ScrollBenchmark {
    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun scrollFeed() = benchmarkRule.measureRepeated(
        packageName = "com.rehan.app",
        metrics = listOf(FrameTimingMetric()),
        iterations = 5,
    ) {
        pressHome()
        startActivityAndWait()
        listRecycler().scrollTo(position = 50)
    }
}

iOS: Xcode Instruments (automate with xcodebuild test)

Leaks, Time Profiler, and energy diagnostics. For CI, use xcodebuild test-without-building with performance tests.

7. Cloud testing & CI integration

Modern teams run mobile tests in every PR. Setup examples:

GitHub Actions + Firebase Test Lab

# .github/workflows/android_test.yml
steps:
  - uses: actions/checkout@v4
  - name: run tests on Firebase
    uses: google-github-actions/firebase-test-lab@v2
    with:
      credentials: ${{ secrets.FIREBASE_SERVICE_ACCOUNT }}
      device: 'model=Pixel6,version=33'
      type: instrumentation
      app: app/build/outputs/apk/debug/app-debug.apk
      test: app/build/outputs/apk/androidTest/debug/app-debug-androidTest.apk

Bitrise for iOS E2E with XCUITest on real devices

Bitrise offers device testing step; also BrowserStack App Automate.

8. Testing edge cases specific to mobile

  • Network: Switch from 5G to 3G, airplane mode, WiFi dropout. Use tools: Facebook Augmented Traffic Control, Charles Proxy, iOS Network Link Conditioner.
  • Permissions: Deny location, camera, then grant later. Test permission rationale dialogs.
  • Deep links & universal links: Verify navigation from web or push notification.
  • Accessibility: TalkBack, VoiceOver, scaling font sizes.
// Example: deep link test with Espresso
@Test
fun testDeepLink() {
    val intent = Intent(Intent.ACTION_VIEW, Uri.parse("myapp://product/123"))
    activityRule.launchActivity(intent)
    onView(withId(R.id.product_detail)).check(matches(isDisplayed()))
}

9. Visual testing & screenshot diff

Catch unintended UI changes across devices. Use Applitools, Snapshot (Android), SnapshotTesting (iOS), or Loki/Chromatic for React Native.

// iOS SnapshotTesting example
assertSnapshot(matching: vc, as: .image(on: .iPhoneSe))

10. Migration from manual to automated culture

Start with high‑ROI journeys (login, purchase, onboarding). Involve QA in code reviews. Aim for 70% passing rate on main branches. Pair manual exploratory with automated regression.

Real‑world case study: fintech app

At a mobile‑first bank, we reduced release testing from 5 days to 6 hours by:

  • Unit test coverage >65% (ViewModel, use cases).
  • 20 critical user journeys automated with Detox on CI (Bitrise).
  • Nightly Firebase Test Lab on 15 physical devices.
  • Percy screenshot diffs for UI regression.

Result: crash‑free rate 99.8%, weekly releases.

Mobile testing roadmap 2024

Immediate adoption: Start writing unit tests for new features. Add a smoke test for your main flow on simulator.

Next 3 months: Integrate a device farm for nightly runs. Set up performance benchmarks.

Long term: On‑device AI testing (like using LLMs to generate test cases), flaky test management, and self‑healing locators.
