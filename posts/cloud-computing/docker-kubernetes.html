Docker & Kubernetes: From Zero to Production-Grade Container Orchestration

By Rehan March 8, 2024 14 min read

‚ö° Containers are the new standard

In the cloud computing era, Docker and Kubernetes have become essential pillars for building, shipping, and running scalable applications. Whether you're deploying microservices or batch jobs, understanding this duo transforms infrastructure complexity into repeatable, automated pipelines. This guide covers everything from containerization fundamentals to production-grade Kubernetes workloads.

üê≥ Docker: Containerization Deep Dive

Why Docker changes everything

Docker packages software into standardized units (containers) that include everything needed to run: code, runtime, system tools, libraries. Works consistently across any Linux/Windows environment.

Essential Dockerfile patterns

# Multi-stage build for minimal production image
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node
EXPOSE 3000
CMD ["node", "dist/server.js"]

Must-know commands & optimizations

  • .dockerignore ‚Äì reduce context size (node_modules, logs, .env)
  • Layer caching: order instructions from least to most frequently changing
  • Distroless / scratch: minimal attack surface for compiled languages

‚ò∏Ô∏è Kubernetes: Production Orchestration

Core concepts ‚Äì Pods, Deployments, Services

Kubernetes schedules and manages containers across clusters. Declarative YAML defines desired state, controllers reconcile.

Stateless Deployment + LoadBalancer

apiVersion: apps/v1
kind: Deployment
metadata:
  name: react-frontend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
      - name: frontend
        image: rehanisongh/frontend:v2.1
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "100m"
          limits:
            memory: "128Mi"
            cpu: "250m"
---
apiVersion: v1
kind: Service
metadata:
  name: frontend-svc
spec:
  type: LoadBalancer
  selector:
    app: frontend
  ports:
  - port: 80
    targetPort: 80

üîÑ CI/CD Integration: GitOps style

Modern pipeline using GitHub Actions + ArgoCD or flux.

# .github/workflows/deploy.yml
name: Build and Push
on: push
jobs:
  docker:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t rehanisongh/app:${{ github.sha }} .
      - run: docker push rehanisongh/app:${{ github.sha }}
      # update k8s manifest with new image (commit to GitOps repo)

ArgoCD syncs automatically from Git repository ‚Üí cluster.

üîê Security best practices (zero-trust)

  • Non-root containers (USER 10001 in Dockerfile)
  • Pod Security Standards (restricted profile)
  • Image scanning (Trivy, Snyk) in CI
  • Network policies (deny all ingress/egress by default)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

üìä Real-world migration: legacy ‚Üí containers

Fintech company reduced infrastructure costs by 47% after moving 200+ microservices to EKS with spot instances. Deployment time dropped from 20 minutes to 2 minutes.

Common pitfalls & how to avoid

  1. Ephemeral storage ‚Äì don't assume local disk persists
  2. Heapster/ metrics-server ‚Äì required for HPA
  3. Pod disruption budgets for availability during updates

üìò Hands-on: Deploy your first app on Kubernetes
