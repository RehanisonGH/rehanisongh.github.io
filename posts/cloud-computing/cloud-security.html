Cloud Security: Zero Trust, Shared Responsibility & Practical Defense

By Rehan February 8, 2025 12 min read

Introduction

As enterprises accelerate cloud adoption, security remains the #1 concern. Unlike traditional data centers, the cloud introduces a shared responsibility model, ephemeral workloads, and identity-first perimeters. This guide ‚Äî tailored for Rehan's Tech Blog under Cloud Computing ‚Äî walks through cloud security essentials, real-world IAM hardening, encryption strategies, and DevSecOps practices that scale. Whether you're on AWS, Azure, or GCP, these principles apply.

1. Shared Responsibility ‚Äì Where cloud ends and you begin

In on-prem, you own everything. In cloud, security of the cloud is the provider‚Äôs job; security in the cloud is yours. Misunderstanding this boundary causes 99% of breaches.

Key demarcations:

  • AWS, Azure, GCP secure hardware, regions, and virtualization layers.
  • Customer secures access, data classification, IAM policies, OS patches (if IaaS), and application code.
  • SaaS like Salesforce shifts more responsibility to vendor, but you still control user access and data sharing.
// Shared responsibility visualized (immutable infrastructure mindset)
# AWS example: S3 bucket policy ‚Äì your responsibility
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:*",
    "Resource": "arn:aws:s3:::critical-bucket/*",
    "Condition": {
      "Bool": {"aws:SecureTransport": "false"}
    }
  }]
}

2. IAM deep dive ‚Äì identity is the new perimeter

Perimeter-based security is dead. In cloud, identity is the control plane.

Zero Trust + least privilege

Start with no access and grant incrementally. Use IAM roles instead of long-lived keys. Implement permission boundaries and policy conditions.

# Terraform ‚Äì enforce MFA and prevent privileged escalation
resource "aws_iam_policy" "restrictive" {
  policy = jsonencode({
    Statement = [{
      Effect = "Deny"
      Action = "*"
      Resource = "*"
      Condition = {
        BoolIfExists : {"aws:MultiFactorAuthPresent" : "false"}
      }
    }]
  })
}

Use just-in-time access (AWS IAM Identity Center, Azure PIM) to avoid standing privileges.

3. Encryption everywhere ‚Äì at rest, in transit, in use

Modern cloud services encrypt by default, but key management differentiates mature orgs.

At rest:

  • S3 SSE-S3, SSE-KMS, or client-side encryption.
  • RDS encrypted storage, EBS volumes.

In transit:

  • TLS 1.2+ enforced; ALB listeners, CloudFront, API Gateway.
  • Service mesh mTLS (Istio, Linkerd).

In use:

  • Confidential computing (AMD SEV-SNP, Intel SGX) ‚Äì keeps data encrypted even during processing.
# Encrypt EBS volume with customer-managed KMS key
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --size 100 \
  --encrypted \
  --kms-key-id alias/ebs-key \
  --volume-type gp3

4. Network security ‚Äì microsegmentation & cloud native firewalls

Don‚Äôt rely solely on security groups. Implement:

  • VPC isolation ‚Äì separate accounts per workload (AWS Organizations, GCP folders).
  • Microsegmentation with AWS Network Firewall, Azure Firewall, or third-party (Palo Alto, Check Point).
  • Private connectivity ‚Äì AWS PrivateLink, Azure Private Link, VPC peering without internet exposure.
// AWS Security Group ‚Äì restrictive SSH (allow only bastion)
resource "aws_security_group_rule" "ssh_bastion" {
  type              = "ingress"
  from_port         = 22
  to_port           = 22
  protocol          = "tcp"
  cidr_blocks       = ["10.0.1.0/24"] # bastion subnet
  security_group_id = aws_security_group.app.id
}

5. DevSecOps ‚Äì shift left, scan everything

Security gates in CI/CD pipelines prevent vulnerable code from reaching production.

Essential scanning:

  • SAST (SonarQube, Checkmarx) ‚Äì static code analysis
  • SCA (Snyk, Dependabot) ‚Äì open source vulnerabilities
  • IaC scanning (Checkov, tfsec) ‚Äì misconfigured Terraform, CloudFormation
  • Container image scanning (Trivy, AWS ECR scanning)
# pre-commit hook example ‚Äì detect hardcoded secrets
pip install detect-secrets
detect-secrets scan --baseline .secrets.baseline

6. Workload & container security

Kubernetes and serverless demand new runtime strategies.

Kubernetes:

  • Pod Security Standards (restricted), OPA/Gatekeeper, Kyverno.
  • Network policies ‚Äì deny all ingress by default.
  • Runtime detection ‚Äì Falco, Aqua, Sysdig.

Serverless:

  • Least privilege Lambda roles; no $LATEST in production.
  • VPC access when handling sensitive data.
  • Third-party tools like PureSec (now Palo Alto).
# Kubernetes network policy ‚Äì deny all non-whitelisted
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

7. Data loss prevention & CASB

DLP for cloud storage, email, and SaaS apps. Use native tools like AWS Macie (PII discovery), Microsoft Purview, or third-party CASB (Netskope, McAfee MVISION).

// Macie job to automatically discover sensitive S3 objects
aws macie2 create-classification-job \
  --job-type ONE_TIME \
  --s3-job-definition bucketDefs=[{bucketId=finance-reports}] \
  --sampling-quantity-included false

8. Incident response in cloud ‚Äì different playbook

Cloud IR is faster, but requires preparation. Use:

  • CloudTrail, VPC Flow Logs, AWS Config for forensic visibility.
  • Automated containment ‚Äì detach IAM roles, quarantine EC2 instances (AWS Systems Manager Automation).
  • Immutable forensics ‚Äì snapshot disks without powering off.
# Isolate compromised instance ‚Äì attach security group with no rules
aws ec2 modify-instance-attribute \
  --instance-id i-0abcdef12345 \
  --groups sg-00000000000

9. Compliance & continuous auditing

Automate compliance mapping (CIS benchmarks, NIST, PCI-DSS). Tools: AWS Security Hub, Azure Security Center, GCP Security Command Center. Enable guardrails ‚Äì SCPs, Azure Policy, Organization policies.

// Service Control Policy ‚Äì deny leaving CloudTrail
{
  "Effect": "Deny",
  "Action": "cloudtrail:DeleteTrail",
  "Resource": "*"
}

10. Emerging threats: AI workloads & supply chain

AI/ML pipelines introduce model theft, poisoned datasets. Secure MLops: protect S3 training data, encrypt model endpoints, scan third-party datasets. Also, software supply chain: attest images (sigstore, cosign), SBOM generation.

# cosign sign container image
cosign sign --key cosign.key ghcr.io/rehanisongh/secure-app:v2

Real‚Äëworld case: Fintech on AWS

A digital bank migrated to EKS and RDS. By implementing:

  • IAM fine-grained roles (no direct user policies)
  • KMS envelope encryption with automatic key rotation
  • VPC endpoints for all API calls (no internet)
  • GuardDuty + Security Hub + automated remediation

They reduced cloud misconfigurations by 82% in six months and passed SOC2 with zero findings.

Best practices ‚Äì concise summary

  1. Enable MFA for every human and break-glass account.
  2. Never use root user for daily tasks.
  3. Centralize logs ‚Äì S3 + Athena, or SIEM (Splunk, Sentinel).
  4. Assume breach ‚Äì design least privilege and segment.
  5. Automate compliance ‚Äì no manual reviews.

Conclusion

Cloud security is not a destination; it's a continuous cycle of identity, detective controls, and automated remediation. The shift to DevSecOps and zero trust is mandatory.
